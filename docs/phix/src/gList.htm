gList

Definition: include xpGUI.e

gdx id = gList(object data, [rtn selected=NULL,] string attributes="", dword_seq args={})
Description: Creates a simple interface element that displays a read-only, non-sortable, header-less list of text-only items, with automatic scrollbars.
NB: scrollbars/scrolling/selection not even started [DEV]

data: (non-optional) a sequence of string [fragment]s or (recommended) a fetch function, see notes and DATA below.
selected: (optional) a procedure which is invoked when the selection changes.
For more information on the attributes and args parameters see gSetAttributes.
This is a paranormalised function. (see technicalia)

Returns: the identifier of the created element.
pwa/p2js: Not yet Supported. [DEV]
See Also: gCanvas, gTable
Example:

(image to be replaced with one that shows a scrollbar, once they are working...)
-- demo\xpGUI\gList.exw
include xpGUI.e

sequence data = {"one",{{"too",XPG_RED,XPG_ITALIC}},"three"}
--                    ^^^ see note below

function get_data(integer idx)
    if idx=0 then return length(data) end if
    if idx=-1 then return 0 end if
    return data[idx]
end function

procedure selected(integer idx)
    ?{"selected",idx,data[idx]}
end procedure

gdx list = gList(get_data,selected),
        -- gList(data,selected), -- (fine too)
    dialog = gDialog(list,"gList",`SIZE=240x80`)
gShow(window)
gMainLoop()
Notes: In some cases it is easier/sensible to specify a (small) fixed set of strings for data, however should that data be frequently (or in fact ever) updated, for instance with diagnostic info or progress messages, it is much better to provide it via a DATA fetch function.

With a fetch function, the application assumes all responsiblity for owning and limiting the data, for example it may be wise to crop a log back to 1,000 entries when adding the 2,001st (ymmv), rather than collect until you inevitably run out of memory. Independently of a gList, that approach should also prevent any ex.err during development potentially taking several minutes to write, and several GB of disk space, as well as using all that memory, on top of gradually getting slower and slower.
It is the responsibility of the gList() to deal with all that quickly and efficiently, whatever you decide to do.
First-class performance is in fact a significant concern in the design of this (deliberately simple) interface element.

Text is always left aligned, with everything in the same font face/pointsize, but optionally scrollable and each line can be specified as a list of coloured and/or normal/bold/italic text fragments.

Should this control not meet your needs then perhaps you should consider a gTable() instead, which can handle multiple sortable columns, with headers, numeric entries, icons[??], etc. Also, a gList is just one half of the original IupList functionality which has been completely redesigned and split, so you may actually be looking for a gDropDown() instead. Should you be looking for a "dropped down with edit area" then what you probably want is actually a gVbox() with a pair of {gText,gList}) children instead, and performing the whole "add entry" thing explicitly yourself, along with setting the gText whenever selection changes in the gList.

While technically a gList is strictly read-only, (with a fetch function) the application is free to make any required changes to the data itself, and that way mimic an editable control, and in fact that is what gEditor [?ged? is planned to do]/does.

Each element of data/return from the DATA fetch function can be a plain string or a sequence of (string | {string,colour[,style]}) fragments.
Note the double braces on the "too" above, and that `,{"t",{"o",XPG_RED},"o"},` (and `,"two",`) would also be valid.
A `,{"too",XPG_RED},` would display "too" in black, then complain XPG_RED is neither string nor {string,colour[,style]},
where colour and style are as per gCanvasText() (including -1 for "as is"). Oh, and when I said black just then, I really meant the gCanvasSetForeground() currenly in force.

Some further examples can be seen in demo\xpGUI\sample.exw [DEV]
Attributes:
? COUNT (read-only, non inheritable) returns the number of items. Note that when a DATA fetch function is in use, and hence the application (rather than the gList) "owns" the data, it simply returns the result from invoking that routine with 0. Should not be confused with ROWS, which concerns the display size.
? DATA Fetch/replace the (static) control-owned data when a DATA fetch function is not in use, which it only is (in use) when this is NULL. You might use this in response to the selection in a set of radio buttons being (manually) altered, but it is not recommended to update the (ever longer) contents this way several times a second. Technically you can also use this to set a handler, and this one is set on creation rather than the other, but subsequently used seamlessly.
? MULTIPLE (creation only) Allows selecting several items simultaneously (multiple list). Default: "NO"/0/false.
To select multiple items, the Ctrl or Shift key must be held down.
Note this changes the required signature of any SELECTED handler.
? ROWS (creation only) The number of lines to be displayed at any given time, if not set the length of data is consulted, minimum 1. Should not be confused with COUNT, which can be much larger.
? SIZE Size of the interface element in pixels, which can be specified in the usual manner as either "wxh", eg "100x150" or (non-string) {w,h}, eg {100,150}. If the width is not set or zero, gGetTextExtent() is used on the longest line, and if the height is not set or zero, the same times ROWS is used.
? TOPITEM (write-only) position the given item at the top of the list or near to make it visible.
? SELECTED (read only) When MULTIPLE is false, returns the index of the currently selected line, or 0 if none.
When MULTIPLE is true, returns a sequence of such, or {} if none.
This can be used as an independant (and on-demand) alternative to a SELECTED handler.
? VALUESTR (read only) Returns data[SELECTED] or "". Note that if data contains any zero-length strings it is not possible to distinguish a SELECTED of zero from any such line, likewise should data contain duplicates it is not possible to use VALUESTR to determine which one is actually selected. This attribute is only intended for use in the "data as sequence" creation mode, and not when a fetch function exists. Should MULTIPLE be true, returns instead a sequence of such, or {} if none.
also ?? CARET, CARETPOS, SCROLLTO,

ACTIVE, CANFOCUS, EXPAND, FONT, MINSIZE, MAXSIZE, TIP, VISIBLE.
Handlers:
DATA A fetch function for retrieving the gList entries/lines/rows.

function fetch_data([gdx id,] integer item)
id: (optional) identifier of the element that activated the event.
item: Index of the desired line, 0 to get the total length, -1 to get the index of the longest.

Naturally, specify a gdx id should you have some use for it, or just omit it otherwise.

When item is 0, as shown above, should simply return the outer/vertical length().
When item is -1, returning 0, or in fact the index of any line that fully fits, simply disables the horizontal scrollbar.
Aside: idx:=fetch_data(-1) is immediately followed by txt:=fetch_data(idx) for all idx>0. Technically speaking, for perfect horizontal scrolling, fetch_data(-1) should effectively invoke gGetTextExtent() on every element of data, and the "data as sequence" creation mode does just that, but in practice it is usually good enough to rely on length(), and in fact when a fixed-width font is in use that perfectly identifies the correct longest line anyway.

Obviously your application may well perform much better by cacheing the results from such [getsize] calls, and you could also of course blatently lie, perform a full scan in the background/idle handler, then gRefresh() and tell the honest truth, iyswim. Obviously, you are perfectly at liberty to break up each data[i] on every fetch, rather than store it in pieces. Should some data[i] contain multiple fragments, it would be down to you to either sum the gGetTextExtent/length or apply said to a contatenation of them (gList does the former on the txt:=fetch_data(idx) mentioned). Naturally if the application owns the data, it can easily limit the gGetTextExtent/length to newly added items only, whereas if the interface element owns the data, it would have to start again from the top whenever given a complete replacement (with no easy way to give it a partial update).

The fetch_data function can simulate a cuebanner by returning 1/1/"Enter xxx" when the internal application data is {}.

The function should return the integer length|idx, a text string, or a sequence of {text|{text,colour[,style]}} fragments.
Only those lines which are actually being displayed are ever requested, plus, optionally, the longest.

Note that gSetHandler(id,"DATA",fetch_data) only works as expected after creation when it was initially NULL, or when a gSetAttribute(id,"DATA",NULL) has also occurred (the latter may need to be 2nd).
[Somewhat unusually there is both a DATA handler and a DATA attribute, with the former only ever used when the latter is NULL.]
 [The latter can also be a (non-null integer) handler, which would be used (in preference to this but) in exactly the same way.]
?SELECTED Action generated when an item in the list is selected. Note there is no "unselect" equivalent (as yet).

procedure selected([gdx id,] integer item) -- when MULTIPLE=NO
-- or --
 procedure selected([gdx id,] sequence items) -- when MULTIPLE=YES

id: (optional) identifier of the element that activated the event.
item: Number of the changed item starting at 1.
items: Sequence of selected items, or {} if none.

Naturally, specify a gdx id should you have some use for it, and just omit it otherwise.

The MULTIPLE attribute determines whether the second parameter is an integer or a sequence, said can also be declared as object and tested within.
also CLICK, DROP, KEY, MOUSEMOVE: All common handlers are supported.
Technicalia As this is a paranormalised function, and with no expectation that you should attempt to memorise this lot, except maybe "dsaa", you can assume that all of the following are potentially perfectly valid, with the fairly obvious restrictions that any parameters must be in strict dsaa-order, and no args without attributes: 

                        gList(data)
                        gList(data, selected)
                        gList(data, attributes)
                        gList(data, attributes, args)
                        gList(data, selected, attributes, args)  -- (the full set)

Note that in practice selected is actually defined as object, and attributes as a sequence rather than a string, but are thoroughly verified to be of the documented types, after being repositioned via xpGUI.e/paranormalise_raa().
